# Django Production-Ready Project

![Pipeline Status](img/pipeline.png)

Реализация безопасной и отказоустойчивой архитектуры Django-приложения с автоматизированным циклом доставки (CI/CD) через GitLab.

## 🚀 Основные фичи архитектуры

### 1. Безопасность (Hardening)
*   **Non-root User**: Контейнер бэкенда запускается от имени `appuser` (UID 1000). Это предотвращает возможность получения root-прав на хосте в случае компрометации приложения.
*   **Изоляция портов**: Порт приложения (8001) работает только во внутренней сети. Доступ осуществляется строго через Nginx как Reverse Proxy.

### Конфигурация для Production
Для перехода из режима разработки в режим сервера было реализовано:
*   **Безопасный DEBUG**: В `settings.py` обеспечен строгий контроль за тем, чтобы на VPS приложение всегда запускалось с `DEBUG = False`.
*   **Настройка ALLOWED_HOSTS**: Список разрешенных доменов ограничен IP-адресом сервера и доменным именем для защиты от атак типа HTTP Host Header.


### 2. Инфраструктура и Хранение
*   **SQLite Persistence**: Использование Docker Volumes для выноса БД на хост (`/home/ubuntu/sqlite_data`). Данные полностью изолированы от жизненного цикла контейнера.
*   **Shared Volumes для статики**: Реализована механика сбора статики (`collectstatic`) из недр контейнера в общую папку на хосте, откуда её забирает Nginx для максимально быстрой раздачи.
*   **Nginx Reverse Proxy**: Контейнер Nginx отвечает за терминацию трафика, защиту и отдачу статических файлов.

### 3. CI/CD Pipeline (GitLab)
Автоматизированный процесс доставки кода, включающий:
*   **Build**: Сборка Docker-образа.
*   **Test**: Автоматический запуск тестов Django перед деплоем.
*   **Safety Deploy**: 
    *   Автоматическое создание снимка (бэкапа) БД перед обновлением.
    *   Обновление через `docker compose up -d --build`.  

### Автоматизированное управление бэкапами 
1. Для минимизации рисков при миграциях в CI/CD интегрирован этап автоматического бэкапа:
`cp db.sqlite3 db.sqlite3.bak_$(date +%F_%H-%M)`
Это обеспечивает гарантированную точку восстановления данных непосредственно перед деплоем новой версии.
Для предотвращения переполнения дискового пространства на VPS реализована ротация бэкапов:
2. Система автоматически анализирует папку бэкапов и оставляет только **10 последних версий**, удаляя устаревшие:
   `ls -t db.sqlite3.bak_* | tail -n +11 | xargs -I {} rm {}`
Это обеспечивает возможность быстрого отката к любой из 10 последних итераций деплоя при полной автономности сервера.

### Команды для восстановления данных из бэкапа:
Если база повреждена, неодходимо выполнить в терминале на сервере:
1. `cd /home/ubuntu12/sqlite_data`
2. `mv db.sqlite3 db.sqlite3.broken`
3. `cp db.sqlite3.bak_YYYY-MM-DD_HH-MM db.sqlite3`
4. `docker compose restart backend`


## 🛠 Особенности реализации (Challenges & Solutions)

### Права доступа 
Одной из ключевых задач была настройка корректной работы **non-root** пользователя с монтированными томами.
*   **Проблема**: При монтировании томов Docker часто назначает владельцем `root`, что блокирует запись для приложения.
*   **Решение**: Синхронизация UID на хосте и внутри Docker-образа. На сервере права на папки `sqlite_data` и `static_root` переданы UID 1000, что позволило сохранить безопасность без использования `sudo` внутри приложения.

### Почему SQLite вместо PostgreSQL? (Architectural Decision)
Для данного проекта был осознанно выбран SQLite3 по следующим причинам:
1.  **Zero-Configuration & Low Overhead**: В условиях небольшого проекта SQLite потребляет на порядки меньше ресурсов (RAM/CPU) по сравнению с инстансом Postgres, что критично при деплое на бюджетные VPS.
2.  **Упрощенный Backup/Restore**: Как показано в моем CI/CD пайплайне, бэкап всей базы данных сводится к одной команде `cp`. Это делает процесс восстановления мгновенным и абсолютно надежным.
3.  **Скорость разработки**: Для текущих нагрузок SQLite полностью закрывает потребности в производительности, избегая лишнего усложнения инфраструктуры (Overengineering).

### Обеспечение безопасности БД
Несмотря на то, что SQLite — это файл, его безопасность в данном проекте обеспечена на нескольких уровнях:
*   **Изоляция в Docker**: Файл базы данных вынесен в отдельный Volume, доступ к которому имеет только контейнер бэкенда. Прямой доступ к папке БД извне закрыт.
*   **File Permissions**: Доступ к файлу на хосте ограничен UID 1000, что предотвращает чтение данных другими системными пользователями.
*   **Pre-deploy Snapshots**: Автоматическая система создания снимков данных перед каждым обновлением кода гарантирует сохранность информации даже при ошибках в миграциях.
